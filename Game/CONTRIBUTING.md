# CONTRIBUTING

本ドキュメントは、実装品質と保守性を維持するための共通規約を定義する。

## 1. 基本原則
- 1つの責務につき1つの関数にまとめる。
- 同種の計算式は共通関数に集約し、複数箇所で重複実装しない。
- 既存コードの命名規則・インデント・波括弧スタイルを維持する。
- 規約は特定機能に依存しない抽象度で記述し、他モジュールにも適用できる形を優先する。

## 2. コメント方針
### 2.1 共通
- 関数の目的と責務が明確に分かるコメントを記述すること。
- 数学計算・座標変換・角度計算・補間処理など、意図が分かりにくい処理には「何を計算しているか」を明示すること。

### 2.2 カメラ関連の必須明記項目
- 注視点（LookAt）の算出方法
- 角度（水平・垂直）と前方ベクトルの関係
- オフセット適用の基準（ワールド基準 / ローカル基準）
- 補間やクランプの意図

## 3. 共通ユーティリティ利用方針
### 3.1 計算ユーティリティ
- 汎用数学・計算ユーティリティは `AppFrame/source/math/mymath.h` に集約する。
- 既存の `mymath` に同等機能がある場合、モジュール側で再実装しない。
- 重複処理（角度正規化、補間、クランプ、水平化、乱数範囲など）を追加で実装する場合は、まず `namespace mymath` への追記を検討する。
- `namespace mymath` へ追記する際は、機能が特定モジュールに依存しないことを確認する。
- 使用時は関数名だけで意図が伝わる命名を優先する（例: `WrapAngle`, `VectorLerp`, `FlattenVector`）。

### 3.2 描画ユーティリティ
- 描画ユーティリティ（デバッグ図形描画、補助線描画、可視化用描画関数等）は `AppFrame/source/draw/mydraw.h` に集約する。

### 3.3 責務分離ルール（計算 / 描画）
- 汎用関数は「計算」と「描画」の責務で分類し、混在させない。
- 数学計算ロジックと描画API呼び出しを同一関数内で同時に扱う必要がある場合は、
  - 計算処理を `mymath` 側へ分離し、
  - 描画側は計算済み結果を受け取って描画のみを担当する構成を優先する。

### 3.4 既存関数の移設ルール
- 呼び出し側の `#include` を移設先に合わせて更新する。
- 旧配置に同名関数を残さず、重複定義を避ける。
- 影響範囲（利用モジュール）を確認し、ビルドエラーがないことを確認する。

## 4. 規約記述のテンプレート
計算処理の規約を追加する場合は、以下の観点で記述する。
- 入力（何を受け取るか）
- 処理（何を計算するか）
- 出力（何を返すか / 何を更新するか）
- 制約（クランプ、正規化、境界値）

機能固有ルールを追加する場合は、「共通原則」と「対象機能の補足」を分離して記述する。

## 5. 規約の継続更新ルール
- レビューや不具合対応で新しいルールが確定した場合、再利用可能な形に一般化して本ファイルへ追記する。
- 追記時は、既存項目との重複を避け、必要に応じて既存項目を統合・整理する。
- 追記時は必ず「目的」「適用範囲」「例外条件（ある場合）」を明記する。
- 規約変更が既存実装へ影響する場合は、影響範囲と移行方針を併記する。
- 一時的な対応や個別事情は恒久ルールにしない。恒久化する場合は再現性と再利用性を確認する。

### 5.1 章分類に基づく追記ルール
- 新しい規約を追加する際は、必ず既存の章構成（1?5章および各節）を確認する。
- 追加内容が既存の章・節に該当する場合は、新規章を作らず該当箇所へ追記する。
- どの既存章にも該当しない場合のみ、新しい章を作成して追記する。
- 新しい章を作成する場合は、章タイトルに適用範囲が分かる名前を付け、関連章との関係（参照先や優先順位）を明記する。
